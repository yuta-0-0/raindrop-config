# Raindrop / Obsidian 自動棚卸しルール v3.1

## 0. 結論

月次棚卸しは、ユーザー確認を前提にしない。
AIがURL・title・excerpt・folder・既存tagsをもとに、必ず1つの最終案を出す。
不明なものも放置せず、決定木に従って `要確認` を付けたうえで暫定分類する。

このルールの目的は、毎回ユーザーがCSVを目視確認しなくても、Raindropのタグ・フォルダ・Obsidian候補が一定品質で整う状態を作ること。

---

## 1. 正本ファイル

分類時は以下の優先順位で参照する。

1. `raindrop_folder_config.md` → フォルダ名・collection ID の正本
2. `raindrop_allowed_tags_v1.md` → 許可タグの正本
3. このファイル（`raindrop_rules_v3_1.md`） → 棚卸し思想・決定木

矛盾がある場合は上位ファイルを優先する。

---

## 2. 基本方針

- Raindropは一次倉庫。
- Makeは日常保存の仮処理。
- チャッピーは月次棚卸しの正本判断。
- Obsidianは選別済み判断資産の倉庫。
- ユーザー確認は必須工程にしない。
- AIは迷っても必ず暫定決定する。
- 迷った場合は `要確認` を付けるが、フォルダ・タグは空にしない。
- CSV再分類では、既存タグを正本にせず、分類材料として扱う。
- `要確認` タグがついていて、かつAIの推奨フォルダと現在のフォルダが異なる場合は、現在のフォルダを維持する（ユーザーが意図的に移動したと見なす）。
- `要確認` タグが外れていてフォルダが変わっている場合も、現在のフォルダを維持する（ユーザーが確認済みと見なす）。
- Haikuが許可タグ外を提案した場合は、その場で使わず、月次棚卸し時に採用判断する。

---

## 3. 月次棚卸しの入力

基本入力：

- Raindrop CSV
- `raindrop_folder_config.md`
- `raindrop_allowed_tags_v1.md`
- このルール

CSV列の優先度：

1. url
2. title
3. excerpt
4. note
5. folder
6. existing tags

本文取得できる場合はURL本文を優先する。
本文取得できない場合は、title / excerpt / url / folder / existing tags だけで暫定分類する。

---

## 4. URL確認ルール

### 4.1 原則

分類時はURL本文確認を優先する。
ただし、全URLの本文確認ができない場合でも処理を止めない。

### 4.2 URL本文確認できた場合

本文内容を最優先して分類する。
既存folderやtagsと本文が矛盾する場合は、本文を優先して修正する。

### 4.3 URL本文確認できない場合

以下の順で判定する。

1. titleに明確な商品名・サービス名・技術名があるか
2. excerptに用途・ジャンル・素材・対象があるか
3. urlドメインから業種や用途が推測できるか
4. 既存folderが大きく外れていないか
5. 既存tagsに検索価値のある名詞があるか

それでも曖昧な場合は、既存folderを維持し、タグに `要確認` を追加する。
ただしタグなしにはしない。

## 5. 自動決定木

### 5.1 まず「SNS投稿URL」を最優先で抜く

URLに以下を含む場合は、本文内容よりURLルールを優先する。

- `x.com/status/` または `twitter.com/status/` → `X`
- `instagram.com/p/` → `Instagram`

SNS個別投稿には必ず `要確認` を付ける。

### 5.2 次に「管理系」を抜く

以下に該当するものは、内容に関わらず管理系を優先する。

- ログイン画面
- 契約管理
- 請求管理
- アカウント設定
- ダッシュボード
- 管理画面

分類：

- サイト運営・保守・解析・ドメイン・サーバー → `サイト管理`
- 外部サービスの契約・請求・ログイン → `アカウント管理`
- WordPress / SWELL 固有 → `SWELL・WordPress`

タグ例：`管理` / `WordPress` / `SWELL`

### 5.3 次に「AI・自動化」を抜く

以下に該当するものは `AI・自動化` を優先する。

- AIツール
- プロンプト設計
- Claude / ChatGPT / Gemini / Codex
- Make.com
- 自動化ワークフロー
- Jina / API連携
- AI分類・AI要約・AI運用

タグ例：`AI` / `管理` / `制作資料`

### 5.4 次に「開発・実装」を抜く

以下に該当するものは `実装・コード` を優先する。

- CSS / JavaScript / DOM / React
- Tauri / Electron
- API / ライブラリ / 実装手順
- GitHub

タグ例：`開発` / `CSS` / `JavaScript` / `実装参考`

### 5.5 次に「素材・デザインリソース」を抜く

以下に該当するものは `素材・フォント` を優先する。

- フォント配布
- アイコン素材 / イラスト素材 / SVG
- テンプレート / デザイン素材 / Illustrator素材

タグ例：`制作資料` / `フォント` / `アイコン` / `イラスト`

### 5.6 次に「デザイン参考・UI」を抜く

以下に該当するものは `デザイン・参考` を優先する。

- Webサイト事例 / LP / UI表現
- レイアウト参考 / タイポグラフィ参考
- 色使い参考 / ビジュアル参考
- ブランド表現 / ポートフォリオ参考

タグ例：`デザイン` / `Webデザイン` / `UI` / `タイポグラフィ` / `デザイン参考`

### 5.7 次に「Mac・アプリ」を抜く

以下に該当するものは `Mac・アプリ` を優先する。

- macOS操作改善 / Macアプリ
- Raycast / Arc / ウィンドウ管理
- システム拡張 / 作業環境改善アプリ

タグ例：`macOS` / `Macアプリ`

### 5.8 次に「物理ガジェット」を抜く

以下に該当するものは `ガジェット` を優先する。

- 電子機器 / デバイス / ケーブル
- キーボード / カメラ機材
- iPhoneアクセサリー / Apple Watchアクセサリー

タグ例：`キーボード` / `カメラ` / `iPhone` / `Apple Watch`

### 5.9 次に「カバン・ポーチ」を抜く

以下に該当するものは `カバン・ポーチ` を優先する。

- バッグ / リュック / ポーチ
- ショルダー / メッセンジャー / トート
- ストラップ単体 / ラップトップポーチ

タグ例：`バッグ` / `PC` / `ナイロン` / `ヌメ革`

### 5.10 次に「ファッション」を抜く

以下に該当するものは `ファッション` を優先する。

- 服 / 靴 / 腕時計 / アクセサリー
- 身につけるもの（カバン・ポーチ以外）

タグ例：`腕時計` / `Apple Watch` / `スニーカー`

### 5.11 次に「ごはん」を抜く

以下に該当するものは `ごはん` を優先する。

- 飲食店 / レシピ / 食べ物
- デザート / ピザ / 地域グルメ

タグ例：`食` / `ピザ` / `デザート` / `東京` / `京都`

### 5.12 次に「DIY・ものづくり」を抜く

以下に該当するものは `DIY・ものづくり` を優先する。

- 材料 / 建材 / 金属加工
- アクリル / 3Dプリンター / 3Dモデル
- 制作方法 / 加工サービス

タグ例：`材料` / `3Dプリンター` / `金属加工` / `アクリル`

### 5.13 最後に「インテリア / 道具」を分ける

#### インテリア

空間を構成するもの。
置いて眺める、空間の印象を作る、家具・照明・収納・置物・生活空間要素。

タグ例：`収納` / `デスク` / `椅子` / `キッチン` / `インダストリアル`

#### 道具

身体の延長として手に持って使うもの。
文房具、カメラ、万年筆、調理道具、作業道具、手元の実用品。

タグ例：`文房具` / `カメラ` / `キッチン`

### 5.14 インテリア / 道具で迷った場合

- 空間に置くことが主目的 → `インテリア`
- 手で使うことが主目的 → `道具`
- 電子機器なら → `ガジェット`
- 制作素材・加工対象なら → `DIY・ものづくり`
- 身につけるなら → `ファッション`
- バッグ・ポーチ系なら → `カバン・ポーチ`

それでも曖昧なら `インテリア` に寄せる。

## 6. タグ付けルール

- タグは必ず `raindrop_allowed_tags_v1.md` の中からのみ選ぶ。
- タグは通常2〜4個、最大5個、最低1個。
- フォルダ名と同じタグは禁止。
- 感想タグ・抽象タグは禁止。
- `要確認` が付いていても、他のタグを必ず併用する。
- Haikuが許可タグ外を提案した場合は月次棚卸し時に採用判断する。

### 自動付与ルール

- SNS投稿URL（x.com/status等）→ 必ず `要確認`
- Google検索URL → 必ず `要確認`
- ネットショップURL → 必ず `ネットショップ`
- 地名は都市・地域レベルまで

### 禁止タグ（新規付与しない）

以下は付与しない。既存CSVにある場合は削除または正規タグへ統合する。

`インテリア` / `ファッション` / `ガジェット` / `ツール` / `ログイン` / `購入検討` / `管理用` / `グッズ` / `役立つ情報` / `参考資料` / `インテリア雑貨` / `かわいい雑貨` / `当日発送` / `B2B` / `資産運用` / `コピペ可能` / `ショップ` / `青モケット` / `IIJmio` / `Type-Cケーブル` / `Web` / `Webアプリ` / `UI効果` / `UI設計` / `用語集` / `デザイン辞書` / `SVGアイコン` / `macアプリ` / `Mac` / `css`

### 正規化ルール

| 旧タグ | 新タグ |
|---|---|
| macアプリ | Macアプリ |
| Mac | macOS |
| css | CSS |
| 管理用 / ログイン | 管理 |
| デザイン辞書 / 用語集 / 参考資料 / リファレンス | 制作資料 |
| SVGアイコン | アイコン |
| カラー・タイポグラフィ | 色 or タイポグラフィ |
| 青モケット | モケット |
| 栃木県 | 栃木 |
| 購入検討 | 削除（必要なら要確認） |
| 資産運用 | 削除 |

## 7. Note欄ルール

Note欄にはAI判定理由を書く。

形式：`[判定理由] 1〜2文。分類根拠と迷いがあればその理由を書く。`

URL本文確認できなかった場合：
`[判定理由] URL本文未確認。title/excerpt/urlから〇〇と判断し、暫定で△△に分類。`

迷いがある場合：
`[判定理由] 〇〇要素と△△要素があるが、主用途が□□のため××に分類。必要なら要確認。`

---

## 8. confidence / 要確認ルール

- **High**：本文またはexcerptで用途が明確。そのまま反映候補。
- **Medium**：title / excerpt / url から十分に推定できる。そのまま反映候補。
- **Low**：本文不足・情報不足・複数カテゴリ混在。`要確認` を付けるが、暫定フォルダ・暫定タグは必ず決める。

`要確認` は作業停止フラグではなく、暫定分類済みの低信頼フラグとして扱う。

---

## 9. Obsidian候補ルール

### 付ける条件

- 制作・開発で再利用できる
- PRISM.blue / MIME.blue / ポートフォリオ / Raindrop / Obsidian運用に接続できる
- デザイン判断の根拠になる
- 実装判断の根拠になる
- 複数回参照しそう
- 自分の言葉でメモを足す価値がある

### 付けない条件

- ただのログインページ
- ただのショップトップ
- 一時的な購入候補
- 情報が薄い商品ページ
- 読み返す目的が薄いURL

### 強制的にObsidian候補にしやすい領域

- 実装・コード
- デザイン・参考
- 素材・フォント
- DIY・ものづくりの技術資料
- AI・自動化のノウハウ記事

## 10. 出力ルール

月次棚卸しでは以下を出す。

1. Raindropインポート候補CSV（主成果物）
2. 棚卸しレポート
3. タグ変更サマリー
4. フォルダ移動サマリー
5. Obsidian候補リスト
6. 低信頼だが暫定決定したリスト
7. 許可タグリストへの追加候補

---

## 11. 最終判断基準

優先順位：

1. 後から検索できること
2. フォルダとタグの責務が分かれていること
3. 迷っても空欄にしないこと
4. 許可タグ外を増やさないこと
5. 月1で続けられること
6. Obsidianをノイズ倉庫にしないこと

迷った場合の最終ルール：

- フォルダは必ず1つ決める
- タグは必ず1〜5個決める
- 不確実なら `要確認` を付ける
- それでも処理は止めない
